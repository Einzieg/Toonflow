# 服务器迁移说明

这份文档是写给服务器上的 Codex 的。

目标是把当前 Toonflow 服务从旧服务器迁移到新服务器，并尽量缩短停机时间。

## 当前已知事实

- 项目路径：`/soft/Toonflow`
- Git 仓库：`git@github-einzieg:Einzieg/Toonflow.git`
- 当前容器名：`toonflow`
- 当前生产端口：`18080`
- 当前持久化目录：`/soft/Toonflow/data`
- 当前 `data` 目录体积约 `11G`
- 当前大头数据在 `data/oss`
- 前端源码已经并入主仓库，不需要单独部署前端仓库
- 当前服务对外访问依赖 `data/web`
- 当前服务运行依赖 `data/serve`
- 当前 COS 已欠费停用，不要在新服务器启用 `OSS_PROVIDER=tencent-cos`

## 重要约束

迁移到新服务器后：

- 不要设置 `OSS_PROVIDER`
- 不要设置任何 `OSS_COS_*` 环境变量
- 必须完整迁移本地 `data/oss`
- 生产启动命令使用 `yarn start`
- 不要使用 Dockerfile 默认的 `yarn dev`

原因：

- 当前代码虽然支持返回 COS URL，但很多生成链路和素材处理链路仍直接读取本地 `data/oss` 文件
- 如果只迁数据库、不迁本地 `oss`，历史图片预览、参考图读取、拼图、视频生成等能力会失败

## 迁移目标

在新服务器上得到与旧服务器等价的服务状态，包括：

- 账号和数据库
- 项目数据
- 历史图片和视频素材
- 技能文件
- 模型目录
- 前端静态文件
- 后端构建产物

## 目录说明

以下目录都在 `/soft/Toonflow/data` 下：

- `db2.sqlite`：主数据库，必须迁移
- `oss/`：历史图片、视频、音频文件，必须迁移
- `vendor/`：供应商配置脚本，必须迁移
- `skills/`：Skill 文件，必须迁移
- `models/`：本地模型目录，建议迁移
- `serve/`：后端构建产物，建议迁移
- `web/`：前端构建产物，建议迁移
- `assets/`：附加素材，建议迁移
- `version.txt`：版本文件，建议迁移

## 迁移策略

使用 `Tailscale + rsync`。

推荐分两轮同步：

1. 旧服务不停机时先做一轮全量同步
2. 切换前停掉旧服务，再做一轮最终增量同步

这样能把停机时间压到最短。

## 前置条件

新服务器需要具备：

- Docker
- Git
- Tailscale
- 可以通过 Tailscale SSH 到旧服务器

先在新服务器确认 Tailscale 联通：

```bash
tailscale status
tailscale ping <旧服务器TailscaleIP>
```

`ping` 结果最好是 `direct`，不是 `relay`。

## 在新服务器执行的迁移步骤

### 1. 准备变量

先把旧服务器 Tailscale IP 填上：

```bash
export OLD_TS_IP="100.x.x.x"
export APP_DIR="/soft/Toonflow"
export RSYNC_RSH='ssh -T -o Compression=no -o StrictHostKeyChecking=accept-new'
```

如果旧服务器不是 `root` 登录，把 `root@` 改成实际用户名。

### 2. 克隆仓库

```bash
mkdir -p /soft
cd /soft

if [ ! -d /soft/Toonflow/.git ]; then
  git clone git@github-einzieg:Einzieg/Toonflow.git Toonflow
fi

mkdir -p /soft/Toonflow/data
```

### 3. 第一轮全量同步

这一步在旧服务不停机时执行。

```bash
rsync -aH --info=progress2 --partial --append-verify \
  -e "$RSYNC_RSH" \
  root@$OLD_TS_IP:/soft/Toonflow/data/ \
  /soft/Toonflow/data/
```

说明：

- `--partial --append-verify` 适合大文件断点续传
- 不要加 `-z`，图片和视频已经压缩过，压缩通常只会更慢

### 4. 在新服务器预构建镜像

```bash
cd /soft/Toonflow
docker build -t toonflow:prod .
```

### 5. 切换前停旧服务器服务

这一步需要在旧服务器执行：

```bash
docker stop toonflow
```

### 6. 第二轮最终增量同步

旧服务停掉后，在新服务器执行：

```bash
rsync -aH --delete --info=progress2 --partial --append-verify \
  -e "$RSYNC_RSH" \
  root@$OLD_TS_IP:/soft/Toonflow/data/ \
  /soft/Toonflow/data/
```

### 7. 启动新服务器容器

注意：这里故意不设置任何 `OSS_PROVIDER` 和 `OSS_COS_*` 环境变量。

```bash
cd /soft/Toonflow

docker rm -f toonflow 2>/dev/null || true

docker run -d \
  --name toonflow \
  --restart unless-stopped \
  -p 18080:18080 \
  -e PORT=18080 \
  -v /soft/Toonflow/data:/app/data \
  toonflow:prod \
  yarn start
```

## 启动后验证

### 1. 看服务是否起来

```bash
docker ps --filter name=toonflow
docker logs --tail 200 toonflow
```

### 2. 验证版本接口

```bash
curl http://127.0.0.1:18080/api/other/getVersion
```

预期：

- 返回 200
- 返回版本号字符串

### 3. 验证关键文件是否存在

```bash
ls -lah /soft/Toonflow/data/db2.sqlite
ls -lah /soft/Toonflow/data/serve/app.js
ls -lah /soft/Toonflow/data/web/index.html
find /soft/Toonflow/data/oss -type f | head
```

### 4. 浏览器验证

打开：

- `http://新服务器IP:18080/web/index.html`

检查：

- 登录页能打开
- 登录后项目列表存在
- 历史分镜图能加载
- 历史衍生资产图能加载
- 进入分镜台时历史素材不报错

## 若新服务器需要重新构建前端和后端

当前仓库已经把前端并入主构建链路。

如果需要重建 `data/web` 和 `data/serve`，执行：

```bash
cd /soft/Toonflow
docker run --rm \
  -v /soft/Toonflow:/app \
  -w /app \
  node:24-bookworm-slim \
  bash -lc 'npm config set registry https://registry.npmmirror.com/ && yarn config set registry https://registry.npmmirror.com/ && yarn install --non-interactive && yarn build'
```

但在本次迁移里，优先直接迁移旧服务器已有的 `data/web` 和 `data/serve`，这样更快、更稳。

## 常见问题

### 1. 页面能开，但图片全部加载失败

排查：

- 是否错误保留了 `OSS_PROVIDER=tencent-cos`
- 是否错误保留了 `OSS_COS_*` 变量
- `data/oss` 是否未同步完整

### 2. 登录后项目在，但生成视频时报素材不存在

通常是：

- `data/oss` 未同步完整
- 只迁了数据库，没有迁本地历史文件

### 3. 服务能起，但访问 `/web/index.html` 404

通常是：

- `data/web` 未迁移
- 容器未挂载 `/soft/Toonflow/data:/app/data`

### 4. `rsync` 很慢

先检查：

```bash
tailscale ping $OLD_TS_IP
```

如果不是 `direct`，速度通常不会理想。

## 回滚方案

如果新服务器验证失败：

1. 停掉新服务器容器
2. 重新启动旧服务器容器
3. 不切换域名或反向代理

旧服务器回滚命令：

```bash
docker start toonflow
```

## 给 Codex 的执行要求

如果你是新服务器上的 Codex，请按下面原则执行：

1. 不要启用 COS。
2. 优先迁移旧服务器已有的 `data`。
3. 先全量同步，再停旧服务，最后增量同步。
4. 生产环境使用 `yarn start`。
5. 验证 `api/other/getVersion`、`/web/index.html`、历史图片加载。
6. 如果某一步失败，保留现场，不要破坏旧服务器数据。
